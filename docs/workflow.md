# Vibe Coding 工作流

## Phase 1: 需求 → PRD

使用 /vibe-coding-prd 或自然语言描述需求。
输出物：结构化 PRD，包含用户故事、功能清单、验收标准。

要求：
- 每个功能有明确的验收标准（可测试的）
- 明确 MVP 范围，标注优先级
- 技术方案建议（但不做最终决策）

## Phase 2: 设计 → DESIGN.md

使用 /web-design 基于 PRD 产出设计规范。
输出物：DESIGN.md（配色、字体、间距、组件规范）+ 代码。

要求：
- 确认设计规范后再开始写代码
- 响应式断点：mobile (< 768px), tablet (768-1024px), desktop (> 1024px)
- 所有设计 token 定义在 tailwind.config 或 CSS 变量中

## Phase 3: 实现 → 原子化交付

子 Agent 分工原则：
- 主 Agent：需求把控、流程编排、验收决策
- 子 Agent：具体开发、单元测试、自测

交付节奏：
1. 拆分功能为可独立交付的最小单元
2. 每个单元：编码 → 自测 → 主 Agent 验收 → git commit
3. commit message 描述 WHAT + WHY，不描述 HOW

质量门禁：
- 每个 commit 前必须通过类型检查（tsc --noEmit）
- 新增组件必须有基本的交互测试
- UI 变更必须在浏览器中确认视觉效果

## Phase 4: 测试 → QA

使用 /qa 或 /qa-only 进行端到端测试。

测试范围：
- 主路径（Happy Path）：核心用户流程
- 边界情况：空状态、极长文本、网络错误
- 响应式：至少检查 mobile 和 desktop 两个断点
- 可访问性：键盘导航、屏幕阅读器基本兼容

## Phase 5: 审查 → Review

使用 /review 进行代码审查，关注：
- 安全性：XSS、注入、敏感信息泄露
- 性能：不必要的重渲染、大列表未虚拟化、图片未优化
- 可维护性：命名清晰、逻辑易懂、无过度抽象
